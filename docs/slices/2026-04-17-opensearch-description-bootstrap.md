# Slice: 2026-04-17 OpenSearch Description Bootstrap

## Selected Pack

- `python_ddd_monolith`

## Runtime Targets

- Python build runtime
- static browser runtime
- filesystem-backed publication output

## Architecture Mode

Deterministic browser-search discovery for the static publication:

- emit one bounded OpenSearch description artifact during the build
- advertise that artifact from the shared HTML head on generated pages
- keep the slice limited to search discovery metadata rather than search
  ranking, indexing, or browser-specific UI behavior

## Discovery Scope

The publication now exposes a complete first-pass static search surface:

- `search.json` is generated as a deterministic static index
- `/search/` exists as a reader-facing search route
- homepage `WebSite` JSON-LD already projects a `SearchAction` targeting
  `/search/?q={search_term_string}`
- the search page now reads and writes the bounded `?q=` contract

One browser-facing discovery gap remains visible in current repository state:

- `build_static_site(...)` emits no OpenSearch description artifact such as
  `/opensearch.xml`
- `_render_document(...)` advertises RSS and manifest metadata, but no
  `<link rel="search" ...>` contract
- generated `dist/` output contains no browser-search description file even
  though the site now has a stable static search route

This slice restores the minimum browser-search discovery surface needed for the
current static publication:

- generate one OpenSearch description artifact at build time
- derive its name, description, and search URL from existing site settings and
  the current `/search/?q=` contract
- advertise that artifact through shared head metadata on generated HTML pages

This slice does not attempt search ranking changes, richer query syntax,
per-browser install behavior, search suggestions, or broader search product
work.

## Use-Case Contract

### `BuildOpenSearchDescription`

Given site settings and the current static search contract, generate
deterministic output for an OpenSearch description artifact such that:

- the artifact exists in the build output without any runtime dependency
- the search template resolves to the existing `/search/?q={searchTerms}`
  route shape
- the artifact remains valid for static hosting on GitHub Pages
- the same repository state yields the same description content

### `RenderSearchAutodiscoveryMetadata`

Given a generated HTML document and the shared head renderer, render
autodiscovery metadata such that:

- representative generated pages advertise one `<link rel="search" ...>`
  entry
- the advertised URL points to the generated OpenSearch description artifact
- existing RSS, manifest, canonical, robots, and social metadata remain
  unchanged

## Main Business Rules

- Browser-search autodiscovery must derive from the existing static search
  route rather than inventing a second search endpoint.
- The OpenSearch artifact must be generated by the build, not maintained as a
  handwritten file in `dist/`.
- The advertised search template should stay aligned with the current bounded
  `?q=` contract on `/search/`.
- The slice stays bounded to search discovery metadata rather than search UI,
  ranking, or analytics.
- GitHub Pages compatibility remains a hard constraint.

## Required Ports

- shared document-head renderer in `src/app/application/use_cases/build_site.py`
- environment reader for site settings
- existing static `/search/` route and `?q=` query contract
- filesystem writer for root-level static build artifacts
