# Slice: 2026-04-17 404 Page Bootstrap

## Selected Pack

- `python_ddd_monolith`

## Runtime Targets

- Python build runtime
- static browser runtime
- filesystem-backed publication output

## Architecture Mode

Deterministic not-found handling for static hosting:

- emit one shared `404.html` artifact during the build
- reuse the existing site chrome and publication language
- keep the slice bounded to one static recovery page

## Discovery Scope

The restored publication now emits homepage, section, topic, saga, archive, and
metadata artifacts, but it still lacks one important static-hosting contract:
a generated `404.html` page for unknown routes.

That gap is visible in current repository artifacts:

- the generated `dist/` output contains no `404.html`
- GitHub Pages remains the primary deployment target, so not-found handling must
  work without an application server

This slice restores the minimum not-found surface needed for the static site:

- generate one root `404.html` page at build time
- explain that the requested page was not found in publication-oriented copy
- point readers back toward the main navigable surfaces

This slice does not attempt redirect logic, search-assisted recovery, custom
status handling beyond static hosting, or analytics around failed routes.

## Use-Case Contract

### `BuildNotFoundPage`

Given site settings and the shared publication frame, generate deterministic
static output for `/404.html` such that:

- the file exists in the build output without any runtime dependency
- readers receive a clear not-found message when a route is missing
- the page links back to stable publication surfaces
- output remains compatible with GitHub Pages hosting

### `RenderNotFoundRecovery`

Given site settings and shared navigation state, render bounded recovery
content such that:

- the page uses the shared site frame and navigation
- the message stays publication-oriented instead of generic framework wording
- recovery links remain valid for the configured static-site base URL

## Main Business Rules

- `404.html` must be generated by the build rather than maintained as a manual
  standalone file in `dist/`.
- The page should reuse existing shared chrome so not-found behavior feels like
  part of the same publication.
- Recovery links should point to stable reader-facing routes such as home,
  archives, sagas, and library.
- The slice stays bounded to one deterministic static not-found page.
- GitHub Pages compatibility remains a hard constraint.

## Required Ports

- shared HTML document renderer for generated routes
- environment reader for site settings
- filesystem writer for static build artifacts
