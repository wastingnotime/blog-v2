# Slice: 2026-04-17 Site Webmanifest Bootstrap

## Selected Pack

- `python_ddd_monolith`

## Runtime Targets

- Python build runtime
- static browser runtime
- filesystem-backed publication output

## Architecture Mode

Deterministic browser-install metadata for the static site:

- emit one bounded `site.webmanifest` artifact during the build
- derive manifest fields from existing site settings and published identity
  assets
- keep the slice limited to install metadata rather than progressive-web-app
  behavior

## Discovery Scope

The publication now emits favicon assets, an apple-touch icon, feed and sitemap
metadata, `robots.txt`, a `404.html`, archives, and a search route. One browser
facing artifact is still missing from the generated output: `site.webmanifest`.

That gap is visible in current repository artifacts:

- the generated `dist/` output includes favicon files but no manifest file
- the earlier identity-asset slice restored icon files only, not the manifest
  that ties them together for browsers

This slice restores the minimum manifest surface needed for the current static
publication:

- generate one `site.webmanifest` file at build time
- reference the existing favicon and touch-icon assets
- derive manifest name values from the configured site settings

This slice does not attempt service workers, offline caching, install prompts,
push notifications, or broader PWA behavior.

## Use-Case Contract

### `BuildSiteWebmanifest`

Given site settings and the published identity-asset set, generate deterministic
static output for `/site.webmanifest` such that:

- the artifact exists in the build output without any runtime dependency
- manifest fields use the configured site title and existing icon assets
- output remains valid for static hosting on GitHub Pages
- the same repository state yields the same manifest content

### `ProjectManifestMetadata`

Given site settings, compute deterministic manifest fields for:

- application name and short name
- start URL and display mode appropriate for the static publication
- icon references to the existing published identity assets

## Main Business Rules

- `site.webmanifest` must be generated from repository-owned settings and assets,
  not maintained as a handwritten output file in `dist/`.
- Manifest icons should reuse the already-published favicon and touch-icon
  assets rather than introducing a new asset family.
- The bootstrap manifest stays bounded to install metadata and identity fields.
- The slice must not imply app-shell or offline behavior that the site does not
  actually provide.
- GitHub Pages compatibility remains a hard constraint.

## Required Ports

- environment reader for site settings
- existing published identity-asset paths
- filesystem writer for static build artifacts
