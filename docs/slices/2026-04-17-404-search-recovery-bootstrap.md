# Slice: 2026-04-17 404 Search Recovery Bootstrap

## Selected Pack

- `python_ddd_monolith`

## Runtime Targets

- Python build runtime
- static browser runtime
- filesystem-backed publication output

## Architecture Mode

Deterministic not-found recovery update for the current publication surface:

- add one Search recovery link to the generated `404.html`
- keep the page aligned with the routes now exposed through the shared chrome
- keep the slice bounded to static recovery affordance only

## Discovery Scope

The publication now has a dedicated `/search/` route and shared navigation
entry, but the static `404.html` recovery page still points readers only toward
home, archives, sagas, and library.

That gap is explicit in current repository artifacts:

- `build_not_found_page(...)` in `build_site.py` renders a fixed recovery list
  with no Search link
- `/search/` now exists as a generated route and reader-facing discovery
  surface
- the earlier 404 slice explicitly treated search-assisted recovery as a later
  follow-on once the search surface matured

This slice restores the minimum search-aware recovery surface needed for the
current publication:

- add one link from `404.html` to `/search/`
- keep existing recovery links intact
- preserve static-only hosting behavior

This slice does not attempt query-prefilled recovery, client-side suggestions,
redirect logic, or broader 404 copy redesign.

## Use-Case Contract

### `BuildNotFoundPage`

Given site settings and the shared publication frame, generate deterministic
static output for `/404.html` such that:

- the page links to the current stable reader-facing recovery surfaces,
  including Search
- output remains fully static and compatible with GitHub Pages hosting
- the same repository state yields the same recovery markup

### `RenderNotFoundRecovery`

Given site settings and current publication routes, render bounded recovery
content such that:

- the page keeps its existing publication-oriented not-found message
- Search appears as one additional recovery option alongside the existing links
- route URLs remain valid for the configured static-site base URL

## Main Business Rules

- `404.html` must stay generated by the build rather than maintained manually
  in `dist/`.
- Recovery links should track the stable reader-facing routes that now exist in
  the publication.
- The slice stays bounded to static recovery affordance rather than dynamic
  search behavior.
- GitHub Pages compatibility remains a hard constraint.

## Required Ports

- shared HTML document renderer for generated routes
- environment reader for site settings
- existing `/search/` static route
